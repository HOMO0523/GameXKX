# 青山镇建筑 JSON 原画与 Tripo 生产设计

日期：2026-07-12
状态：用户已确认设计，等待书面规格复核

## 1. 目标

把青山镇当前抽象代理建筑替换为可复现、可审核、适合 Tripo 单图建模的正式建筑资产。所有正式 Tripo 输入原画必须由 JSON 驱动生成，不能依赖聊天中的临时提示。建筑族采用“共享风格母版 + 每栋独立资产 JSON”，统一画法但保持明显的结构差异。

Tripo 07-11 22:18 生成、约 50,745 四边面的橙色屋顶小民居所使用的真实输入图，是本流程的首个已验证 Tripo 构图参考。此前视觉伴侣中的 SVG 仅为结构说明，不属于正式风格参考，也不得进入生产链路。

## 2. 文件与数据权威

### 2.1 共享风格母版

`SourceAssets/TownPCG/QingshanBuildings/style/QS_InkToon_Building_v2.json` 负责建筑族共同视觉语言：

- 中国水墨卡通、干刷粗墨线、两到三个明度层级。
- 5%–10% 的受控歪扭；歪扭发生在屋脊、檐口、墙体和粗梁柱的大轮廓。
- 粗梁柱、宽门窗、暖黄色宣纸窗。
- 屋顶以完整大色块表达，不逐片画瓦。
- 米白无纹理背景、固定三分之四视角、完整接地面。
- 禁止镜像对称、黑洞窗、密集格栅、三层柱头、连续小斗拱、碎木片、可读文字、商标和水印。

### 2.2 单栋资产 JSON

每栋 `asset.json` 负责自身参数：

- 资产 ID、功能、尺寸、层数、占地、入口和前向轴。
- 独有轮廓、屋顶形式与颜色、门窗数量和位置。
- 主体、侧翼、塔楼、门廊、雨棚和平台的拆件关系。
- 招牌、灯笼等独立附件的 Socket、Pivot 和父建筑 ID。
- GPT 原画提示、生成版本、审核状态、图片路径与 SHA-256。
- Tripo 高精度生成、四边面重拓扑目标、贴图阶段和 UE 导入状态。

共享母版只定义“属于同一个游戏世界”，单栋 JSON 定义“为什么不是同一栋房子”。任何正式图片都必须能追溯到共享母版版本和单栋 JSON 版本。

## 3. 六类建筑族

### 3.1 最大商铺 L-A

现有 Tripo 商铺继续作为城镇尺度与风格锚点。它是宽体两层、组合屋顶和开放柜台构成的唯一大型市场地标。主体不重新设计成另一种画法；后续只补充符合新 JSON 契约的独立附件和登记信息。

### 3.2 中型客栈 M-A

不规则 L 形两层主体，暖红主屋顶、青绿侧翼屋顶，入口偏心。主体、侧翼、门廊或小塔楼分别生成，避免大型建筑一次单图生成造成融合、穿插和背面错误。

### 3.3 沿街商铺 M-B

一层半宽体建筑，横向展开，深檐压低。正面具有清楚的开放柜台和独立雨棚，使用墨绿或靛青主屋顶。轮廓、入口和立面组织必须与客栈和民居不同。

### 3.4 小型民居 S-A

采用已在 Tripo 验证的紧凑两层构图：橙色弯屋顶、粗双开门、粗梁柱和宽框窗。正式重生版本把黑色窗面改成暖黄色宣纸窗，并保持完整三分之四视角和干净米白背景。

### 3.5 小型工坊 S-B

低矮长体、靛青屋顶、大尺寸工作门，侧面带独立棚架或烟囱模块。整体重心低且横向，不允许通过给民居换屋顶颜色来冒充工坊。

### 3.6 河岸小屋 S-C

一层或一层半，赭黄小屋顶，架高石基或木平台，临河一侧有宽开口。主体、架高平台和码头连接件分别生成，保证 PCG、桥岸和码头接口可控。

每一类至少改变高宽比例、屋顶方向、入口、门窗组织和附属体中的三项。屋顶颜色只作为辅助差异，不能充当主要变化。

## 4. 原画构图与拆件契约

- 每张图只出现一个建筑主体或一个独立模块。
- 固定三分之四视角，同时显示建筑正面和一个侧面。
- 完整显示屋顶、墙脚和接地面，不裁切。
- 米白无纹理背景，不出现地面场景、人物、植物、其他建筑或环境装饰。
- 图片内不写标注；身份、尺寸、用途和模块关系由文件名与 JSON 表达。
- 黄色宣纸窗必须是清楚的暖黄色实体色块，不能画成黑洞或透明空洞。
- 不画逐片瓦、密集窗格、重复小斗拱和碎木件。
- 招牌、灯笼和雨棚不焊死在主体原画中，必须拥有独立原画和资产 ID。

默认拆分：

- 小民居、沿街商铺、工坊：主体一张；招牌、灯笼、雨棚分别单图。
- 中型客栈：主体、侧翼、门廊三类图；确有需要时增加独立小塔楼。
- 最大商铺：沿用现有主体；新增附件独立生成。
- 河岸小屋：主体、架高平台、码头连接件分别生成。

## 5. JSON 到 Tripo 的门控链路

```text
共享风格 JSON + 单栋 asset.json
              ↓ 结构与字段校验
        GPT 生成单体原画
              ↓ 视觉与构图检查
 图片路径、版本和 SHA-256 回写 JSON
              ↓ 原画审核通过
         Tripo 高精度模型
              ↓
      约 50,000 四边面重拓扑
              ↓
          Tripo 贴图生成
              ↓
 UE 5.8 StaticMesh + 统一 BSDF 卡渲材质
```

未通过原画审核时不得消耗 Tripo 高精度生成。重拓扑未完成时不得开始最终贴图。未经 JSON 登记和哈希锁定的图片不得上传 Tripo。

## 6. 原画验收

每张原画必须同时满足：

- 单体完整且没有裁切，墙脚和接地面清晰。
- 在 128 px 缩略图中仍可辨认功能和主要轮廓。
- 正面入口、至少一个侧面、屋顶结构清楚。
- 暖黄色宣纸窗没有黑洞感。
- 大轮廓存在受控不对称，但没有融化、坍塌或无法解释的受力结构。
- 主体最多三个主要体块、五组次级结构信息。
- 没有细瓦、密格窗、三层柱头、碎木噪点、文字或环境干扰。
- 与建筑族其他成员至少有三项结构差异。

失败图片保留版本记录但不能写入 `approved_tripo_input`。一次迭代只修正一个主要问题，避免同时改变画法、比例和结构。

## 7. Tripo 与 UE 验收

- Tripo 使用高精度模型，不使用低面智能网格作为正式建筑。
- 正式建筑以约 50,000 四边面为首轮重拓扑目标；附件单独计算，不塞入主体预算。
- 检查屋顶、门窗、墙脚和背面是否闭合，避免悬浮、融合和薄片穿插。
- 重拓扑模型必须保持单一稳定接地平面、底部中心 Pivot 和 JSON 声明的前向轴。
- 贴图在重拓扑后生成；窗口宣纸、墙、木构、屋顶和石基必须能映射到 UE 材质语义。
- UE 5.8 使用统一 BSDF 卡渲材质。首轮不以降低源模型质量换性能；运行时通过 Nanite、LOD、PCG 实例、附件复用和距离裁剪收敛。
- 正式模型先替换隔离 B1 场景中的稳定资产 ID，不直接覆盖原始 `L_QingshanInn`、B0R、角色、PaperZD 或用户手调相机。

## 8. 首个黄金样板

首个生产对象是 `BLD_QS_S_A_HOUSE`：

1. 把已验证 Tripo 输入图的构图、尺寸和轮廓转换为完整 `asset.json`。
2. 引用 `QS_InkToon_Building_v2.json`，把黑窗修正为暖黄色宣纸窗。
3. 由 JSON 重新生成一张正式原画，不直接把旧图改名当成 JSON 产物。
4. 通过原画验收后记录文件路径、生成版本和 SHA-256。
5. 用户确认黄金样板后，才批量派生其余建筑 JSON 与原画。

## 9. 完成标准

- 共享风格母版和六栋独立资产 JSON 均通过结构校验。
- 所有 Tripo 输入图可从 JSON 重现并拥有版本与 SHA-256。
- 六类建筑在轮廓和功能上明显不同，同时保持统一水墨卡通画法。
- 大型建筑和复用附件按契约拆分，没有把标注或文字画进主体图。
- Tripo 高精度、约 50,000 四边面重拓扑、贴图和 UE 导入均有状态记录。
- B1 隔离场景完成代理替换，同时原始地图和受保护资产保持不变。
