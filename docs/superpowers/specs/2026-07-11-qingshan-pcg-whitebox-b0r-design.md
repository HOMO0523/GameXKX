# 青山镇 PCG 白盒与氛围关键帧 B0R 修订设计

日期：2026-07-11
状态：交互设计已确认，等待书面规格复核

## 1. 目标

先在 Unreal Engine 5.8 中建立一个可重复生成、可步行检查、可从固定机位截图的青山镇白盒，再以真实白盒截图为结构约束，分别制作四张氛围关键1
帧。白盒必须先解决城镇布局、道路与河流关系、建筑组团、地形起伏、外围环山和玩家视线，不在本阶段制作正式模型、贴图、Paper2D 动画或 Substrate Toon BSDF 材质。

本修订采用方案 B：

```text
结构化 PCG 布局数据
→ UE 独立白盒关卡
→ 四个固定机位截图
→ 四张图分别风格化与审核
→ 全部通过后合成氛围总览
```

生成模型不得重新发明桥、道路、河流或建筑位置。UE 白盒截图是每张氛围关键帧的空间结构来源，`REF_QS_ENV_STYLE_LOCK` v003 只提供色彩、笔触和水墨 Toon 语言。

## 2. 修订关系与保护边界

本规格建立新的修订批次 `B0R`，不重置、覆盖或删除现有 Batch 0 的生成次数、文件、哈希和审核记录。

- `REF_QS_ENV_STYLE_LOCK` v003 保留为风格参考；其中错误的桥结构、碎石和零散地面笔触不得传递到新输出。
- `REF_QS_SCALE_LINEUP` v003、`REF_QS_CAMERA_ROUTE` v002 和 `REF_QS_SURFACE_PALETTE` v002 保留为历史审核记录，不作为 B0R 的生产结构输入。
- 原 Batch 0、Batch 1、Batch 2、Batch 3 的锁定状态保持不变。B0R 未通过前，不允许 Tripo、正式 Paper2D、正式材质或 UE 正式资产生产。
- 旧版《青山镇 QuickRoad / PCG 生成链路设计》中的“北门位于北侧中央、主街南北贯穿”被本 B0R 白盒修订替代。旧文档保留用于审计，不静默改写历史。
- 不保存或覆盖 `/Game/GameXXK/Maps/L_QingshanInn`，不移动现有北门 F、任务 NPC、角色、PaperZD、相机或用户手调 HD2D 平面。
- 不修改用户当前的 `Config/DefaultEditor.ini`。

白盒使用独立关卡：

```text
/Game/GameXXK/Maps/Dev/L_Qingshan_PCG_Whitebox_B0R
```

只有白盒和四张关键帧通过审核后，才另行设计如何把布局安全地集成进正式关卡。

## 3. 空间范围与方向

白盒采用不规则山谷聚落，而不是棋盘式或中轴对称城镇。

- 可游玩镇区目标范围约为 `300m × 240m`。
- 含外围山体的整体范围约为 `450m × 380m`。
- 北门 F 在地图右侧，是固定玩法锚点。
- 角色初始视线不直接正对北门；从玩家视角能识别北门位于右前或右侧。
- 从北门进入的主路先向左弯，再向镇中心回转，形成可读但不笔直的进镇动线。
- 河流斜穿场地，并至少形成两个可读弯道。
- 最大建筑是核心地标，但偏离画面和地图几何中心，并相对道路斜置。
- 南侧码头位于较低地势，通过坡道或小路连接核心街区。
- 外围山体形成环山包围感，但北门方向必须保留明确谷口和出口视线。

主要玩家动线为：

```text
地图右侧北门 F
      ↘ 弧形主路
   核心建筑群 ─ 主桥
          ↘ 小路下坡
            南侧码头
```

## 4. 白盒组成

### 4.1 固定玩法锚点

白盒必须登记但不得随机化以下锚点：

- `NorthGateFAnchor`：北门 F 的位置、朝向和交互保护区。
- `TownCoreAnchor`：核心建筑群的构图中心，不等同于地图中心。
- `MainBridgeAnchor`：主桥中心、桥面方向和两侧桥头连接区。
- `SouthDockAnchor`：南侧码头平台和到达路径。
- `QuestNPCExclusion`：现有任务 NPC 的保护区；独立白盒可用代理体表示，不复制或修改其逻辑。

### 4.2 道路

- 一条由北门进入的弧形主路，白盒宽度约 `7–8m`。
- 两至三条 `3.5–5m` 宽支路，分别连接建筑组团、主桥和南侧码头。
- 道路不得构成等间距十字网格，也不得长距离笔直穿过画面中心。
- 建筑入口只朝向可达道路或院落，不使用随机旋转后再强行贴路。
- 优先使用已经验证可用的 QuickRoad 5.8 道路样条和找平能力。若插件接口在白盒阶段不可用，使用 UE Spline 和代理路面完成同一数据契约，不阻塞空间审核；正式集成前仍需单独通过 QuickRoad 验证。

### 4.3 河流与桥

- 河流使用独立 River Spline 和白盒河槽，不把 QuickRoad 的内部 `RiverTool` 当作正式河流生成器。
- 河流宽度可沿样条变化，主桥区域保持可验证的桥岸净空。
- 主桥放在主路或主要支路与河流的真实交汇处，桥中心线与道路连续。
- 第二处过河点只允许小桥或简易渡口，用于形成路线选择，不与主桥争夺视觉层级。
- 桥必须是独立固定锚点，不由随机散布节点生成。

### 4.4 建筑组团

首轮数量上限：

| 类型 | 数量 |
|---|---:|
| 最大地标建筑 | 1 |
| 中型建筑 | 4–6 |
| 小型建筑 | 8–12 |

组团规则：

- 最大建筑体量最强，但位置偏心，朝向与道路形成斜角。
- 中小建筑围绕道路、院落和高差形成三至四个小组团，不沿道路等距排队。
- 同一组团内至少同时改变两项：占地大小、层高、朝向、前后退距或地台高度。
- 保留一至两块可呼吸的空地，用于集市、院落或视线展开，不用建筑填满全部候选地块。
- 白盒建筑只表达占地、层高、屋顶大轮廓、入口方向和附属侧翼，不表达门窗、瓦片或装饰细节。

### 4.5 地形、外围山体与植被代理

- 地形由缓坡、台地、河岸下降和码头低地组成，不使用整张水平平面。
- 建筑地块允许局部找平，但各组团之间保留可读高差。
- 外围使用 `20–30` 个低模山体或岩壁代理块形成不均匀环山轮廓。
- 山体高度、宽度、前后层级和旋转必须变化；禁止重复模型等距围成圆环。
- 北门谷口保持畅通，核心机位不得被近山完全遮挡。
- 树丛首轮约 `100` 组，只使用低成本实例化体块验证疏密、遮挡和框景。
- Paper2D 植物、小道具和风摆动画不在白盒阶段生产。

## 5. 四个固定机位

四个机位必须保存准确 Transform、视场角、目标点和截图分辨率。每个机位输出独立的 16:9 白盒截图，不先制作四格拼图。

### 5.1 `CAM_QS_GATE_ARRIVAL`

- 展示北门位于画面右侧或右前方。
- 角色的主要移动方向沿弧形道路向左进入城镇，而不是正对北门。
- 中景可预告核心建筑或远处山谷，但不能一次暴露全部城镇。

### 5.2 `CAM_QS_TOWN_CORE`

- 最大建筑是视觉主角，但不位于正中央。
- 至少能读到一组前景小建筑、一组中景建筑和后景山体。
- 小建筑通过高低、前后和旋转形成错落，不能像模型展台一样整齐排列。

### 5.3 `CAM_QS_MAIN_BRIDGE`

- 清楚表现道路如何上桥、桥如何跨河、桥头如何接入建筑或植被区域。
- 河流必须从画面中穿过而不是只露出一小块装饰水面。
- 桥的结构来自白盒，不允许后续图像生成改变桥孔、桥向或连接关系。

### 5.4 `CAM_QS_SOUTH_DOCK`

- 展示从镇区下坡到码头的空间关系。
- 码头、岸线、低地植被和远处建筑形成层次。
- 不增加未经批准的船只系统、第二大型地标或现代港口设施。

## 6. B0R 数据契约

白盒布局必须保存为结构化 JSON 或等价的确定性数据文件，而不是只存在于关卡摆放结果中。至少包含：

```text
schema_version
revision_id = B0R
source_map
whitebox_map
coordinate_reference
map_bounds
playable_bounds
random_seed
fixed_anchors
road_splines
river_splines
building_plots
terrain_zones
terrain_masks
foliage_zones
exclusion_zones
spawn_caps
cameras
quickroad_status
```

每个锚点、样条控制点、地块和相机必须有稳定 ID。坐标统一使用 UE 厘米，文档展示可以同时给出米制说明。固定随机种子必须得到一致的候选地块、建筑类别选择和植被代理分布。

Surface 不再作为氛围概念图承担布局职责。后续单独建立 Surface Assignment Contract，记录：

```text
ZoneType → UE 组件 → 材质族 → 混合方式
```

预期分工为：Landscape 负责土、草和河岸基础层；QuickRoad 负责道路表面；Water Mesh 负责河面；RVT 或遮罩负责接缝；山石使用独立 Mesh 材质。B0R 白盒只登记区域，不制作正式材质。

## 7. 全类别中度夸张规则

B0R 后续氛围关键帧采用“全类别一起中度夸张”，不是只把商铺画得夸张。

- 建筑：放大屋顶、深檐、侧翼和高低差，减少碎窗格与细瓦片。
- 桥与码头：强化单一主弧、桥面厚度、桩和平台剪影，不堆小石块。
- 树木：使用两至三个大冠团和清楚树干，不逐叶表现。
- 山体与岩石：强化倾斜、峰体比例和大块阴影，不用细碎皴法覆盖全表面。
- 小道具：只保留一眼能读懂的主轮廓，不形成杂物堆。
- 地表与水体：使用大色块和少量方向性笔触，不把微型纹理当作细节质量。

夸张强度为 B：轮廓和比例明显偏离写实，但道路宽度、入口尺度、桥梁通行和地形关系仍然可信。

## 8. 氛围关键帧生产与审核

B0R 新建独立记录，不挪用已耗尽或接近耗尽的旧 REF 版本额度。四个机位分别具有自己的资产 ID、提示、版本、文件哈希和审批状态。

每张关键帧的输入固定为：

1. 对应 UE 白盒截图：锁定空间、透视和遮挡关系。
2. `REF_QS_ENV_STYLE_LOCK` v003：只锁定色彩、线条和笔触。
3. B0R 全类别中度夸张规则：锁定造型语言。

每张图单独生成、单独退回、单独批准。生成时禁止改变固定锚点、增加建筑数量、移动桥河关系或把机位改成远景明信片。四张都通过后，才允许制作一张不计入结构输入的组合审核图。

## 9. 性能边界

白盒阶段的目标是低成本验证结构，而不是模拟最终画质。

- 只使用基础几何、低模代理、Spline 和 ISM/HISM。
- 建筑代理不启用复杂碰撞，不使用正式 2K/4K 贴图。
- 树丛和重复山体不得为每个实例创建独立高成本 Actor。
- 首轮不超过 19 栋建筑代理、2 个过河点、100 组树丛代理和 30 个外围山体代理。
- PCG 默认编辑器内生成并保存结果，不在运行时持续重建整座城镇。
- 数量上限、随机种子、裁剪距离和样条分段必须集中记录，不写成无法追踪的散落常量。

正式资产面数、Nanite、贴图和 Toon BSDF 性能预算沿用后续生产规格，不在白盒阶段提前承诺或验证。

## 10. 失败与回退

- UE MCP 不可用时停止关卡写入，不强制关闭可能含未保存修改的编辑器。
- QuickRoad 不可用时只回退为兼容相同数据契约的代理 Spline，不修改正式关卡，也不宣称 QuickRoad 链路已验证。
- 地块与道路、河流、桥头、北门或交互保护区冲突时，丢弃地块，不缩放建筑到不可读比例。
- 主桥无法连续连接两岸道路时，白盒验收失败；不得用装饰物遮盖连接错误。
- 固定随机种子重跑产生不同结果时，先修复确定性问题，不继续制作关键帧。
- 任一机位缺乏清楚前中后景时，先调整白盒布局或机位，不让图像生成模型替代空间设计。
- 所有操作限定在 B0R 目录和独立白盒关卡。任何集成到正式关卡的动作需要新的明确审批。

## 11. 白盒验收

### 11.1 布局与玩法

- 北门 F 位于地图右侧，交互保护区未被道路、建筑或植被侵占。
- 玩家能理解进镇方向，但初始视线不直接正对北门。
- 弧形主路、支路、主桥和码头形成连续可达路线。
- 最大建筑明显但偏心，中小建筑在大小、高度、角度和退距上形成错落。
- 道路不存在棋盘式长直线和等距建筑排布。
- 河流斜穿场地并形成可读弯道，主桥准确连接两岸道路。
- 地形具有缓坡、台地、河岸下降和码头低地；建筑基座没有明显悬空或深插。
- 外围山体形成环山感，同时保留北门谷口和关键视线。

### 11.2 四个机位

- 四张 16:9 白盒截图分别具有清楚的前景、中景和后景。
- 北门、核心建筑群、主桥和码头各自是对应机位的明确主题。
- 俯视调试图能够读懂整体道路、河流、地块、排除区和山体边界。
- 近景画面不空，远景画面不把城镇压缩成不可读的小模型集合。

### 11.3 数据与稳定性

- B0R 布局数据包含稳定 ID、边界、样条、锚点、地块、排除区、相机和数量上限。
- 相同随机种子连续生成至少两次，关键布局和实例统计一致。
- 保存并重新打开白盒关卡后，四个相机 Transform 和白盒结果保持一致。
- B0R 的写入不包含 `L_QingshanInn`、`DefaultEditor.ini` 或用户手调资产。

## 12. 后续顺序

本规格通过书面复核后，下一步才编写实施计划。实施顺序固定为：

1. 核对 UE 5.8 编辑器、项目 MCP 和 QuickRoad 可用状态。
2. 创建 B0R 数据文件和独立白盒关卡。
3. 建立固定锚点、地形范围、道路、河流和桥。
4. 建立建筑地块、代理建筑、外围山体和树丛代理。
5. 保存四个固定机位并导出白盒截图与俯视调试图。
6. 完成白盒确定性、路线、构图和写入范围验收。
7. 用户批准白盒后，分别风格化四张氛围关键帧。
8. 四张关键帧全部批准后，才建立组合氛围总览并讨论正式资产生产。

在第 6 步通过前，不执行图像生成；在第 8 步通过前，不解锁 Tripo、正式 Paper2D 或正式 UE 材质生产。
